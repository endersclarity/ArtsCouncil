**CLI interface:**


**Section 1: Umami API calls** (base URL: `https://cloud.umami.is/analytics/us/api/websites/${WEBSITE_ID}`)

Add a `umamiGet(path, params)` helper that:
- Constructs URL with query params
- Sets `Authorization: Bearer ${UMAMI_BEARER_TOKEN}` header
- On 401 response: throw error with message "UMAMI_BEARER_TOKEN expired. Refresh from cloud.umami.is dashboard: JSON.parse(localStorage.getItem('umami.auth')).token"
- On other errors: throw with status code and body text
- If `--verbose`: log each API call URL to stderr

API calls (in order):
1. `GET /stats?startAt={ms}&endAt={ms}` → overview stats (visitors, visits, pageviews, bounces, totaltime)
2. `GET /metrics?startAt={ms}&endAt={ms}&type=event` → event name counts
3. `GET /event-data/properties?startAt={ms}&endAt={ms}` → discover all event+property combos
4. For each discovered combo: `GET /event-data/values?startAt={ms}&endAt={ms}&event={name}&propertyName={prop}` → property value breakdowns
5. `GET /metrics?startAt={ms}&endAt={ms}&type=url` → page view breakdown

**Section 2: Supabase API call**

Fetch chat logs:
```javascript
const url = new URL(`${SUPABASE_URL}/rest/v1/chat_logs`);
url.searchParams.set('select', 'id,created_at,session_hash,query_text,response_text,intent,assets_referenced,duration_ms,model');
url.searchParams.set('created_at', `gte.${startIso}`);
url.searchParams.set('created_at', `lte.${endIso}`);
url.searchParams.set('order', 'created_at.desc');
```
Headers: `apikey` and `Authorization` both set to `SUPABASE_SECRET_KEY`.

**IMPORTANT:** Supabase URLSearchParams with duplicate keys (two `created_at` params) may not work correctly. Use the PostgREST `and` syntax or construct the URL string manually:
```
?select=...&created_at=gte.${startIso}&created_at=lte.${endIso}&order=created_at.desc
```
Test this — if URLSearchParams overwrites duplicate keys, build the query string manually with `&`.

**Section 3: Intent classification** (implement the decision tree from PIPELINE-ARCHITECTURE.md Section 3)

Create `classifyIntentClusters(eventsSummary, eventProperties, chatLogs)`:
- Index event counts by name from `eventsSummary`
- Index property values from `eventProperties` (keyed as `eventName.propertyName`)
- Apply the 7-type decision tree in priority order:
  1. `trip_researcher` — if `outbound:lodging-vrbo` > 0 → HIGH confidence
  2. `tonight_planner` — if `outbound:event-ticket` > 0 AND `events:date-filter.filter=tonight` > 0 → HIGH
  3. `tonight_planner` — if `events:date-filter.filter=tonight` > 0 AND chatbot dining query exists → HIGH
  4. `event_hunter` — if `outbound:event-ticket` > 0 (without tonight) → MODERATE
  5. `art_seeker` — if 2+ of [Galleries, Cultural Organizations, Performing Arts, Museums] in `category:filter.category` → MODERATE
  6. `family_planner` — if `category:filter.category` includes Education OR chat queries match /kids|family|children/i → MODERATE
  7. `event_hunter` — if `toggle:events-14d.state=on` + `detail:open` count >= 2 → MODERATE
  8. `local_explorer` — if `toggle:open-now.state=on` AND overview.pageviews <= 5 → LOW
  9. `casual_browser` — residual → LOW

Each cluster: `{ confidence, evidence: [...], estimated_visitor_count }`. Estimated visitor count is a rough inference: assign known high-confidence clusters 1 visitor each, subtract from total, distribute remainder.

**Section 4: Chatbot query micro-intent classification**

For each chat log, classify `query_text.toLowerCase()` into micro-intents via keyword matching:
- dining: /eat|drink|dinner|restaurant|bar|brunch|lunch|cafe|coffee/
- lodging: /hotel|stay|lodging|night|vrbo|airbnb|cabin|camp/
- outdoor: /hike|trail|outdoor|nature|park|swim|bike|kayak/
- arts: /gallery|art|studio|museum|exhibit/
- same_day: /tonight|this evening|now|today/
- future_trip: /weekend|saturday|sunday|next week|visit/
- family: /kids|family|children|child/
- live_music: /music|concert|show|live|band|gig/

Multiple labels can apply. Store as `query_intents` array on each log entry.

**Section 5: Business referral attribution scoring**

Create `scoreVenueAttribution(eventProperties, chatLogs)`:
- For each venue mentioned in `detail:open.name`, `outbound:event-ticket.venue`, `outbound:website.venue`, `marker:click.name`:
  - Count signals per venue from each property breakdown
  - Parse chat response_text for `[[Venue Name|pid]]` patterns to count chat_recommendations
  - Apply scoring formula:
    ```
    score = (outbound_ticket_clicks * 100) + (outbound_website_clicks * 40) + (detail_opens * 10) + (marker_clicks * 5) + (chat_recommendations * 5) + (chat_click_throughs * 30)
    ```
  - `chat_click_throughs` comes from `chat:deeplink-click.venue` property values (only available after 07-01 ships)
  - Assign tier: score >= 100 → tier 1, score >= 10 → tier 2, else tier 3
  - Add `note` field for venues with chat_recommendations > 0 but chat_click_throughs == 0: "Score may undercount — chatbot click-through data depends on Phase 7 instrumentation"
- Sort by attribution_score DESC
- Return top 20 (or all if fewer)

**Section 6: Demand signals summary** (human-readable for committee)

Create `buildDemandSignals(eventProperties, venueAttribution, chatLogs, queryIntents)`:
- `top_categories`: from `category:filter.category` sorted by count
- `top_venues_by_engagement`: from `detail:open.name` sorted by count, top 10
- `top_outbound_referrals`: from venue_attribution tier 1 entries
- `chatbot_topics`: aggregate micro-intent counts across all chat logs
- `chatbot_venue_recs`: from [[Venue|pid]] regex extraction, sorted by count
- `zero_result_searches`: from `search:zero.query` values (or `explore:search` where results_count=0)
- `notable_patterns`: generate 2-3 human-readable strings describing behavioral clusters (e.g., "Tonight-planning cluster: event ticket click-through + dining chatbot query suggests same-day visitor")

Create `.github/workflows/demand-signal-report.yml` following the design in PIPELINE-ARCHITECTURE.md Section 5.

```yaml
name: Monthly Demand Signal Report

on:
  schedule:
    - cron: '0 9 1 * *'   # 9am UTC on the 1st of every month
  workflow_dispatch:       # Allow manual trigger from Actions UI

jobs:
  report:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Run demand signal pull
        env:
          UMAMI_WEBSITE_ID: ${{ secrets.UMAMI_WEBSITE_ID }}
          UMAMI_BEARER_TOKEN: ${{ secrets.UMAMI_BEARER_TOKEN }}
          SUPABASE_URL: ${{ secrets.SUPABASE_URL }}
          SUPABASE_SECRET_KEY: ${{ secrets.SUPABASE_SECRET_KEY }}
        run: |
          # Calculate previous month date range
          MONTH_START=$(date -d "$(date +%Y-%m-01) -1 month" +%Y-%m-01)
          MONTH_END=$(date -d "$(date +%Y-%m-01) -1 day" +%Y-%m-%d)
          MONTH_LABEL=$(date -d "$MONTH_START" +%Y-%m)

          echo "Generating report for $MONTH_START to $MONTH_END"

          mkdir -p reports
          node scripts/demand-signal-pull.mjs \
            --from "$MONTH_START" --to "$MONTH_END" \
            --out "reports/${MONTH_LABEL}-demand-signals.json" \
            --verbose

      - name: Commit report
        run: |
          git config user.email "reports@exploregvnc.com"
          git config user.name "Demand Signal Bot"
          git add reports/
          git diff --staged --quiet || git commit -m "chore: monthly demand signal report $(date -d '$(date +%Y-%m-01) -1 month' +%Y-%m)"
          git push
```

**Key notes:**
- Uses `workflow_dispatch` so it can be triggered manually from the GitHub Actions UI
- The script runs with `--verbose` in CI for debugging
- Creates `reports/` directory if it doesn't exist
- Only commits+pushes if there are actual changes (the `git diff --staged --quiet ||` guard)
- Does NOT install npm dependencies (script uses native fetch, zero deps)
- Secrets must be configured in GitHub repo settings: UMAMI_WEBSITE_ID, UMAMI_BEARER_TOKEN, SUPABASE_URL, SUPABASE_SECRET_KEY

Also create `reports/.gitkeep` to ensure the reports directory is tracked.

Add a comment header in the YAML documenting the quarterly UMAMI_BEARER_TOKEN refresh requirement:

```yaml
# Then update the UMAMI_BEARER_TOKEN secret in this repo's Settings → Secrets.
```
