## Accomplishments

- S2: events14dMode filter+sort now uses pre-built `dataIndexMap` for O(1) lookups — eliminates O(n^2 log n) performance issue on slower devices
- S6: `getStore` removed from `window.CulturalMapDreamboardModel` public API — internal helper stays internal

- S1: `renderEventBookmarkButton(title, date, size, venue, layer)` now accepts venue + layer params and emits `data-event-venue` / `data-event-layer` attributes — trip builder now has venue context on saved events

## Decisions Made

- `dataIndexMap` keyed by asset name (`d.n`) — names are unique in data.json, safe as map key
- `getStore` remains as an internal function; only removed from the `window.CulturalMapDreamboardModel` export object — all internal callers unchanged
- `getModel()` caches the real module once `model.hasPlace` is truthy, re-checks on each call until then
- `renderEventBookmarkButton` venue/layer params default to empty string via `String(undefined || '') = ''` — fully backward compatible; callers that don't pass them get empty attrs
