# Accessibility audit — v1-discovery-map main map surface

Source-level, read-only. Files: `index.html` (150 lines), `styles.css` (3756 lines), `app.js` (~3867
lines). WCAG 2.1 AA pragmatic pass. No files edited. Browser tool verification NOT performed here —
see "Needs live verification" at the end.

Ranks: **A0** blocks use · **A1** serious barrier · **A2** friction · **A3** nit.

---

## A0 — Blocks use

### A0-1. Selection drawer close destroys focus, no restoration
`app.js` `closeSelectionDrawer()` (~1200-1212) removes `.open` and replaces
`els.detail.innerHTML`, but never calls `.focus()` afterward. If the close button (or any element
inside the drawer) held focus, the `innerHTML` swap deletes that node mid-interaction and focus
silently falls back to `<body>`. Every keyboard/screen-reader user loses their place on every single
place/event selection close — this is the single highest-impact bug in the file because it fires
constantly (core interaction loop: pick a place → view detail → close).
**Fix:** capture the triggering element (or a sensible fallback like the previously active rail
card / place-list row) before the innerHTML swap, and call `.focus()` on it (or on the drawer's own
close button before it's destroyed, falling back to the trigger) after the swap.

### A0-2. Mode tabs are `role="tablist"` wrapping plain buttons — broken ARIA tab pattern
`index.html:66-71` — `<div class="mode-tabs" role="tablist" ...>` wraps four `<button class="mode-tab" data-mode="...">` with no `role="tab"`, no `aria-selected`, no `aria-controls`. `app.js`
`syncModeTabs()` (~1195-1198) only toggles the `.active` CSS class, never touches ARIA state. A
screen reader announces "tab list" and then reads plain unlabeled buttons with no indication which
mode is active or that Left/Right-arrow tab-list conventions apply — this actively misleads AT users
about the widget they're in (worse than having no role at all).
**Fix:** either add `role="tab"` + `aria-selected` (synced in `syncModeTabs()`) + `aria-controls`
pointing at the panel that changes, or drop `role="tablist"` from the wrapper and let the buttons be
a plain `role="group"` of toggle buttons with `aria-pressed`.

## A1 — Serious barrier

### A1-1. Rail track re-render also drops focus
`app.js` `renderDiscoveryRail()` (~3073-3100) does a full `innerHTML` replace of `#rail-track` on
every rail-chip click and every drift toggle. If a rail card was keyboard-focused (via the working
ArrowLeft/Right walker at ~3110-3124), that focus is destroyed with no restoration — same class of
bug as A0-1, slightly lower severity because it's not the primary interaction loop.
**Fix:** before re-render, note which card (by id/index) was focused; after re-render, re-focus the
equivalent card, or the first card, or the triggering chip.

### A1-2. No focus trap / no Escape-to-close on the selection drawer
Confirmed only one Escape listener exists in the entire file (`app.js:3069-3071`), and it is scoped
to `drift.on` only — ending the drift tour. There is **no** Escape handler for the selection drawer
and no Escape handler for the mobile nav menu. Combined with A0-1, this means once a keyboard user
opens the drawer, there is no keyboard-native way to dismiss it except tabbing to the close button
and activating it — and doing so loses focus per A0-1. No focus trap exists either, so on layouts
where the drawer visually covers the panel (mobile), Tab can walk out of the "modal-ish" drawer into
obscured background content.
**Fix:** add a document-level (or drawer-scoped) Escape handler that closes the selection drawer
when open, and add a lightweight focus trap (or at minimum `inert`/`aria-hidden` on background
content) while the drawer is open on narrow viewports.
*Correction to task brief*: the brief assumed multiple competing Escape listeners; the fork's sweep
found only one (drift-scoped). No fighting-handler risk exists today — the gap is an absent handler,
not a conflicting one.

### A1-3. Red-on-white text fails 4.5:1 in at least two live (non-hover) instances
Computed contrast: `#ff2500` on `#ffffff` = **3.81:1** (fails AA 4.5:1 for normal text; would need
18.66px+ bold or 24px+ to qualify as "large text" at only 3:1).
- `styles.css:3509-3510` — `.trail-facts dt { font-size: 11px; font-weight: 700; ... color: var(--ncac-red); }` — red 11px bold label text on white detail-card background. Real content text (trail fact labels: distance, difficulty, etc.), not a hover/decorative state. **Fails.**
- `styles.css:3319-3320` — `.featured-muse-badge-link { font-size: 12px; font-weight: 800; color: var(--ncac-red); }` — red 12px bold link text on white. **Fails.**
- Also present but lower-stakes because they're `:hover`/`:active`-only states (not the resting
  state, and typically paired with an underline/border change as a second cue):
  `styles.css:699-701` `.rail-reset-chip:hover`, `styles.css:3240-3241` `.featured-muse-chip:hover`,
  `styles.css:3286-3287` `a.muse-directory-badge-link:hover`.
- The codebase is already self-aware of one instance of this problem — `styles.css:423-425` has a
  comment on `.mode-tab.active` explicitly noting "13px text fail contrast (white on #FF2E00 is
  3.7:1)" and choosing ink/white instead — but that awareness wasn't applied to the trail-facts and
  muse-badge-link instances above.
**Fix:** darken the red for text use (e.g. a ~4.5:1-passing red text token distinct from the
decorative `--ncac-red`), or pair red text with `--ink` weight/underline and drop red from the text
color for body-sized labels.

### A1-4. Drift bar auto-updates an `aria-live="polite"` region every ~8.2s indefinitely, twice per cycle
`index.html:115` — `<div class="drift-bar" ... aria-live="polite" ...>`. `app.js` `updateDriftBar()`
(~2905-2920) fires twice per `DRIFT_DWELL_MS` cycle (~2874) — once at flight-start (en-route/dimmed
state) and once at arrival (full name/meta swap) — with no `aria-atomic="true"` and no
debounce/throttle. For a tour with no fixed end, this reads as a screen reader announcing new
place-name + meta + count roughly every 4 seconds effectively (two announcements per ~8.2s stop),
continuously, for as long as Drift runs. Likely perceived as spam and drowns out the user's ability
to do anything else with their screen reader meanwhile.
**Fix:** only announce on arrival (drop or silence the en-route intermediate update from the live
region, e.g. move the en-route dimming to a non-live child), add `aria-atomic="true"` so partial DOM
diffs don't cause fragmented reads, and consider `aria-live="polite"` → an explicit user-triggered
announcement model if drift is meant to run long unattended.

## A2 — Friction

### A2-1. Desktop/fine-pointer touch targets remain under 44px (known, half-fixed)
`styles.css:3372-3383` and `3494-3499` fix `.rail-chip`, `.mode-tab`, `.filter-chip`, and others to
`min-height: 44px` **only inside `@media (pointer: coarse)`**. The CSS's own comment
(`styles.css:3491-3493`) states the un-fixed baseline: "rail chips 31px, mode tabs 33.5px" — same
numbers the June 2026 critique flagged. On fine-pointer devices (desktop, and critically
**touchscreen laptops / 2-in-1s that report `pointer: fine` when a mouse is also present**, or
hybrid input Windows tablets), these controls stay under the 44×44 CSS px target size guidance.
`.rail-chip` padding is `6px 13px` at `font-size: 12px` (~30px computed height); `.mode-tab` padding
is `8px 6px 7px` at `font-size: 13px` (~33-34px). This is AA-adjacent (WCAG 2.1 AA Success Criterion
2.5.5 Target Size is actually AAA-only in 2.1; 2.5.8 Target Size Minimum at 24px is the AA 2.2
criterion) — both controls clear the stricter 24px AA floor but miss the 44px AAA/mobile-usability
guidance the codebase itself is chasing.
**Fix (optional, already partially prioritized by the team):** decide whether pointer:coarse-only
scoping is intentional (likely yes, to preserve desktop density) — if so this is a non-blocking nit;
flagging in case the intent was full 44px coverage.

### A2-2. Map markers / place selection may be keyboard-unreachable via the map itself
No `tabindex` assignment to MapLibre marker DOM nodes was found in app.js — marker interaction
appears mouse/touch-only, which is a common MapLibre GL limitation. This is likely non-blocking
*if* every place reachable via a marker is also reachable via the places list or discovery rail
(both of which are keyboard-operable per the fork's sweep) — needs live verification to confirm no
map-only content exists.

### A2-3. MUSE story thumbnails use `alt=""`
`app.js:1666, 1698` — MUSE story thumbnail images consistently render `alt=""`. Defensible if
adjacent visible title text always duplicates the image's content (decorative treatment), but worth
confirming intent wasn't "forgot to write alt text" versus a deliberate decorative call.

## A3 — Nit

### A3-1. `aria-pressed` pattern is inconsistent, not absent
`outing-row`, `outing-pill`, `featured-muse-chip`, and `#drift-toggle` all correctly carry and sync
`aria-pressed` (`app.js:1223, 1238, 1257`, `syncDriftChrome()` ~2922-2929) — good pattern. It's just
not extended to `.mode-tab` (see A0-2), which is the actual gap, not a missing pattern.

### A3-2. `--ink-soft` and legend text pass comfortably
`--ink-soft: #555555` on white = 7.46:1, on `--paper-warm #f7f7f7` = 6.96:1 — both clear AA (4.5:1)
with margin at all sizes checked (legend 12px, place-list meta 11px, marker-preview 11px, etc.). No
action needed; confirms the muted-gray token choice itself is sound — only the red-as-text token is
the contrast risk (see A1-3).

### A3-3. Drift pulse dot and other continuous CSS animation correctly gated
`.drift-live` pulse (`styles.css:3661-3671`) is covered by the global
`@media (prefers-reduced-motion: reduce)` block (`styles.css:3207-3220`, plus a scoped rule at
3743-3750). JS-side, `prefersReducedMotion()` and `gsapMotionOn()` gate all camera flights, drift
orbit rotation, rail scroll behavior, and GSAP tweens per the fork's sweep — no ungated animation
found in either file. This area is in good shape; no fix needed.

## Needs live verification (browser/AT required — not confirmable from source alone)

1. **A0-1 / A1-1 focus loss** — confirm in a real screen reader (NVDA/VoiceOver) and via visible
   focus ring: does focus actually land on `<body>` (or vanish) after closing the drawer / after a
   rail re-render, as the source strongly suggests?
2. **A0-2 tab pattern** — confirm what NVDA/VoiceOver/JAWS actually announce when landing on the
   `role="tablist"` wrapper with plain buttons inside (announcement wording varies by AT and may be
   more or less confusing than predicted from source).
3. **A1-2 drawer trap** — on a narrow/mobile viewport, physically Tab through the page with the
   drawer open and confirm whether focus visually escapes into obscured background content.
4. **A1-4 drift live-region spam** — run an actual screen reader through one full ~8.2s-cadence
   drift cycle and confirm whether the double-announcement is as disruptive as the source pattern
   suggests, or whether AT-specific coalescing softens it.
5. **A2-1 touch targets** — measure actual rendered `.rail-chip`/`.mode-tab` box height in a real
   fine-pointer browser (Windows touch laptop reporting `pointer: fine`) to confirm the CSS
   media-query gap is reachable in practice, not just in theory.
6. **A2-2 map-only content** — click through every marker on the live map and confirm each
   corresponding place is also reachable via keyboard through the places list / rail (no map-only
   places).
7. **Color contrast rendering** — spot-check A1-3's computed sRGB contrast against the actual
   rendered pixels (font smoothing / anti-aliasing can shift perceived contrast slightly, though not
   enough to change pass/fail here).
